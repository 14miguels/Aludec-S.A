# Diário de Estágio — 13/03/2026

## Objetivo do dia
Iniciar o desenvolvimento do projeto integrador na Aludec: compreender a estrutura das fichas técnicas (SDS) e definir a abordagem para extrair automaticamente informação relevante de PDFs para posterior preenchimento dos ficheiros Excel (SEVESO / REACH).

## Trabalho realizado

### 1. Organização do projeto
- Criação do repositório e definição da estrutura inicial de pastas.
- Criação da documentação base: `data-schema.md` e `pdf-mapping.md`.
- Definição inicial do schema de extração de dados (campos a extrair dos PDFs).

### 2. Análise dos documentos
- Análise de exemplos de Safety Data Sheets (SDS) de diferentes fornecedores.
- Identificação das secções relevantes (ex.: Secção 1 — identificação; Secção 3 — composição; secções de classificação e propriedades).
- Verificação das diferenças de layout entre fornecedores e impacto nos campos necessários para SEVESO / REACH.

### 3. Implementação inicial de extração
- Instalação e teste da biblioteca PyMuPDF (fitz) para leitura de PDFs.
- Criação do módulo `pdf_reader.py` com funcionalidades para:
  - abrir um PDF;
  - contar páginas;
  - extrair texto página a página;
  - guardar o texto bruto em `output/raw_text.txt` para análise.

### 4. Pré-processamento e separação de secções
- Implementação de limpeza de texto (remoção de linhas repetidas, normalização).
- Desenvolvimento de um section splitter (`parsing/section_splitter.py`) que identifica secções do SDS (SECÇÃO 1..16) e produz um dicionário com mapeamento "Secção X" → texto da secção.

### 5. Parser da Secção 3 (composição)
- Criação de `parsing/section3_parser.py` para extrair componentes químicos:
  - uso de expressões regulares para deteção de números CAS;
  - lógica para localizar o nome da substância (procura em linhas anteriores quando necessário);
  - filtragem de metadados (CE, REACH, índice, classificações H, percentagens).
- Resultado atual: extração de pares como { name: "xileno", cas: "1330-20-7" }.

## Problemas encontrados
- A extração de texto nem sempre preserva a estrutura visual (tabelas «desmontam»).
- A localização de campos varia entre fornecedores; o nome da substância nem sempre aparece imediatamente antes do CAS.

Soluções provisórias adotadas:
- Subir várias linhas para procurar nomes válidos quando o CAS é identificado.
- Criar parsers tolerantes a variações de layout.

## Observações
- As fichas SDS seguem normalmente o formato padrão de 16 secções, mas a estrutura interna varia entre fornecedores, exigindo parsers mais robustos.

## Próximos passos
- Melhorar o parser da Secção 3 para cobrir mais variações (capturar percentagens e classificações de perigo).
- Consolidar a estrutura de saída em JSON.
- Mapear os campos extraídos para o formato exigido pelos ficheiros Excel (SEVESO / REACH).
- Criar um dataset inicial de PDFs anotados para testes e validação.
