

# Diário de Estágio — 20/03/2026

## Objetivo do dia
Evoluir o protótipo de extração automática de informação de fichas de dados de segurança (SDS) em PDF, passando de uma abordagem baseada em parsing manual/regex para uma arquitetura modular suportada por LLM, com foco na identificação de secções do documento e na extração de substâncias da Secção 3.

## Trabalho realizado

### 1. Reorganização da arquitetura do projeto
Ao longo do dia foi revista a estrutura do projeto, com o objetivo de separar melhor responsabilidades e evitar mistura entre parsing, extração, schema e interação com modelos de linguagem.

Foi definida uma arquitetura modular com os seguintes blocos principais:
- **`schema/`** para definição das estruturas de dados do sistema;
- **`parsing/`** para leitura e limpeza de texto, bem como fallback heurístico;
- **`llm/`** para interação com o modelo, prompts e extração semântica;
- **`pipeline/`** para orquestração do fluxo completo.

Esta reorganização permitiu clarificar o papel de cada componente e reduzir acoplamento entre módulos.

### 2. Definição do schema de dados
Foi criado o ficheiro de schema com classes de dados para representar a estrutura final do sistema. Foram definidas classes como:
- `Section`
- `Substance`
- `Hazard`
- `ProcessingMetadata`
- `SDSDocument`

A opção por dataclasses permitiu modelar os dados de forma simples e consistente, sem necessidade de lógica adicional nas classes.

### 3. Implementação do leitor de PDF
Foi ajustado o módulo responsável pela leitura de PDFs, passando a existir uma única função para:
- receber o caminho de um PDF;
- extrair o texto de todas as páginas;
- devolver o texto bruto como string.

Foi abandonada a abordagem intermédia de guardar o texto em ficheiro físico, permitindo integrar diretamente a leitura do PDF no pipeline.

### 4. Separação entre parsing e extração
Foi consolidada a distinção entre:
- **parsing**: preparação do texto e fallback por regras;
- **extração**: interpretação semântica do conteúdo com apoio de LLM.

Nesta fase tornou-se claro que uma abordagem puramente baseada em regex e verificação linha a linha não seria escalável, tendo em conta a existência de cerca de 20 templates diferentes e múltiplas línguas.

### 5. Implementação da camada LLM
Foi iniciada a integração com o modelo **Gemini Flash 2.5**, escolhido nesta fase por questões de custo e rapidez de prototipagem.

Foram criados módulos específicos para:
- configurar a ligação ao modelo (`client.py`);
- armazenar os prompts (`prompts.py`);
- dividir o documento em secções (`section_splitter.py`);
- extrair substâncias da Secção 3 (`section3_extractor.py`);
- converter respostas JSON em objetos Python (`parser.py`).

Foi também configurado o uso de `.env` para guardar a API key de forma separada do código.

### 6. Construção de prompts para splitting e extração
Foram construídos dois prompts principais:
- um para identificação e separação das secções do documento;
- outro para extração das substâncias da Secção 3.

Posteriormente, para acelerar os testes, o prompt de extração foi ajustado para permitir trabalhar diretamente sobre o documento completo, pedindo ao modelo para:
1. identificar a Secção 3;
2. extrair apenas as substâncias listadas nessa secção.

Esta alteração permitiu testar mais rapidamente a robustez do extractor sem depender totalmente da qualidade do section splitter.

### 7. Robustez do parser
Foi reforçada a robustez do parser que interpreta as respostas do LLM, passando a lidar com:
- respostas vazias;
- JSON inválido;
- listas ou dicionários com chaves como `sections` ou `substances`;
- itens inválidos no interior da lista;
- campos ausentes ou `None`.

O objetivo foi garantir que o sistema não falha perante respostas imperfeitas do modelo, devolvendo listas vazias ou ignorando entradas inválidas em vez de crashar.

### 8. Construção do pipeline principal
Foi criado o pipeline principal do sistema, responsável por executar o fluxo completo:
1. ler o PDF;
2. limpar o texto;
3. tentar divisão em secções com LLM;
4. usar fallback se necessário;
5. identificar a Secção 3;
6. extrair substâncias com LLM;
7. usar fallback heurístico em caso de falha.

Foram também melhoradas as mensagens de execução no terminal, tornando o processo mais fácil de seguir em debugging.

### 9. Testes realizados
Foram realizados testes com pelo menos dois documentos:

#### PDF em português
O pipeline conseguiu:
- identificar corretamente a Secção 3;
- extrair a maioria das substâncias;
- associar corretamente nomes, números CAS e percentagens na maioria dos casos.

Um dos testes revelou ainda uma falha pontual na associação entre CAS e nome da substância, o que confirmou as limitações da abordagem heurística anterior.

#### PDF em espanhol com layout diferente
Foi testado um segundo documento, em espanhol e com formato distinto do primeiro. O sistema conseguiu extrair corretamente um conjunto alargado de substâncias, incluindo números CAS e intervalos de percentagem.

O resultado demonstrou que a abordagem com LLM apresenta capacidade real de generalização a diferentes línguas e templates.

Persistiram apenas alguns erros pontuais, nomeadamente:
- um nome vazio para uma substância com CAS identificado;
- uma percentagem extraída incorretamente num dos compostos;
- dificuldades associadas à leitura de tabelas partidas e colunas desalinhadas.

Ainda assim, o comportamento global foi considerado bastante positivo para esta fase do protótipo.

### 10. Nova extensão proposta ao sistema
Durante o dia foi ainda proposta uma evolução adicional do sistema: passar a analisar também PDFs relativos a **misturas de tintas, solventes e vernizes**, com protocolos de preparação e utilização mais complexos.

Esta nova vertente implicará não apenas extração de informação, mas também:
- interpretação de instruções técnicas;
- análise de protocolos;
- realização de cálculos de conversão;
- tratamento de documentos sem template fixo.

Esta proposta aumenta significativamente a complexidade do projeto e aponta para uma evolução do sistema de uma ferramenta de extração para uma ferramenta mais avançada de análise técnica assistida.

## Dificuldades sentidas
- Estruturação inicial do projeto e separação correta entre módulos;
- Gestão de imports e packages em Python (`ModuleNotFoundError`, necessidade de `__init__.py` e execução com `python -m`);
- Falhas na construção de prompts usando `.format(...)` com JSON embutido;
- Necessidade de tornar o parser resistente a respostas imperfeitas do LLM;
- Latência e peso do section splitting com o documento completo;
- Dificuldade em extrair corretamente informação de tabelas com linhas partidas;
- Necessidade de conciliar rapidez de prototipagem com robustez arquitetural.

## Conclusão do dia
O trabalho realizado permitiu passar de uma abordagem frágil baseada em parsing manual para um protótipo funcional suportado por LLM, com pipeline modular e capacidade real de generalização a documentos de formatos e línguas diferentes.

O sistema já demonstra capacidade para extrair, de forma automática, substâncias químicas com respetivo nome, número CAS e percentagem, a partir de SDS em PDF. Embora ainda existam limitações na leitura de tabelas mais complexas, os resultados obtidos validam a abordagem seguida.

A proposta de extensão do sistema para análise de misturas de tintas, solventes e vernizes abre uma nova fase do projeto, com maior exigência ao nível da interpretação semântica e do cálculo técnico, reforçando a relevância de uma arquitetura baseada em LLM.
