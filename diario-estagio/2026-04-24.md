

# Diário de Estágio — 24/04/2026

Hoje foi um dia de progresso significativo no desenvolvimento do sistema de extração e exportação de dados das fichas de dados de segurança (SDS), com foco na consolidação do pipeline completo até ao ficheiro Excel final.

Inicialmente, trabalhei na melhoria da qualidade dos dados extraídos, garantindo que a informação proveniente do LLM ficava mais limpa e consistente. Foi necessário ajustar o tratamento de campos como a designação da substância e a referência do produto, removendo ruído, códigos indevidos ou identificadores incorretos. Em particular, foi consolidada a regra de que a referência deve corresponder ao código do produto e a designação deve corresponder apenas ao nome comercial do produto.

Durante os testes, foi possível perceber melhor a diferença entre problemas causados pelo código e problemas causados pela própria qualidade dos PDFs. Alguns documentos apresentavam texto desformatado, caracteres separados e blocos mal extraídos, o que afetava diretamente a qualidade da resposta do LLM. Esta análise permitiu confirmar que o pipeline funciona melhor quando recebe texto bem estruturado e que será necessário manter mecanismos de normalização e validação para lidar com PDFs de pior qualidade.

Um dos principais desafios do dia foi a normalização e o alinhamento dos dados no processo de exportação para Excel. Embora os dados estivessem corretamente extraídos e presentes no JSON, verificou-se que algumas colunas estavam desalinhadas, levando a que valores como percentagens, números CAS e números CE aparecessem nas colunas erradas. Este problema foi identificado como sendo causado por inconsistências entre o número de colunas definido nos cabeçalhos do Excel e os dados efetivamente inseridos em cada linha. Após análise e debugging, foi corrigido o alinhamento, garantindo correspondência exata entre a estrutura esperada e o conteúdo exportado.

Outro ponto importante foi a integração do número CE (N.º CE), que inicialmente não estava a aparecer no Excel. Apesar de este valor já estar presente na Secção 3 da SDS e no output JSON produzido pelo LLM, o campo perdia-se durante o processo de normalização/exportação. Para resolver este problema, foram ajustados o prompt de extração da Secção 3, o parser das substâncias e o verificador de substâncias, passando o campo `ce_number` a ser transportado juntamente com `name`, `cas_number` e `percentage`. Foi ainda implementado um mecanismo de recuperação no exportador, de forma a preservar o número CE a partir dos dados brutos quando necessário.

Também foram feitas melhorias na robustez da base de dados e do exportador. Foram adicionadas colunas opcionais para dados operacionais, como consumo anual e capacidade de armazenamento, e garantido que estas colunas são criadas automaticamente quando o exportador é executado isoladamente. Foi também necessário tratar corretamente objetos `sqlite3.Row`, evitando erros ao tentar aceder aos dados como se fossem dicionários normais.

Foram introduzidas regras de cálculo para valores que não são diretamente extraídos da SDS, nomeadamente COV (%) por regra de negócio, COV total em toneladas e quantidade máxima armazenada em toneladas. No entanto, ficou claro que alguns destes cálculos dependem de dados operacionais internos, como o consumo anual e a capacidade de armazenamento, que não constam normalmente das SDS. Assim, ficou definido que estes dados terão de ser integrados posteriormente através de uma fonte externa ou tabela própria, e não através do LLM.

No final do dia, o pipeline ficou bastante mais sólido, conseguindo:

- extrair corretamente substâncias da Secção 3;
- extrair percentagens, números CAS e números CE;
- extrair perigos, transporte, propriedades físico-químicas e categoria Seveso;
- normalizar referência e designação do produto;
- guardar os dados na base de dados;
- exportar para Excel com as colunas corretamente alinhadas;
- produzir resultados muito próximos do formato esperado.

Ficou identificado como próximo passo a integração de dados operacionais internos, necessários para preencher automaticamente campos como consumo anual, capacidade de armazenamento, COV total e quantidade máxima armazenada. Esta parte será tratada numa fase seguinte, uma vez que estes valores não pertencem diretamente à ficha de dados de segurança.

De forma geral, o progresso foi bastante positivo. Foram resolvidos problemas críticos de integração entre LLM, normalização, base de dados e exportação para Excel, aproximando o projeto de uma versão funcional e demonstrável.
